### 0.1 The vision — what this project is

**Guildr is a LAN-only PWA that lets one operator stand up a project, hand it to a Qwen-powered SDLC pipeline, and *review / follow / intervene* as it runs — with the touch surface feeling like an incredible game, not a spreadsheet dashboard.**

Load-bearing constraints — not preferences, not negotiable:

1. **Qwen-first, local-first.** PRIMARY (192.168.1.13) + ALIEN (192.168.1.70) llama-server endpoints. **128 KiB context ceiling is a hard wall.** Any design that casually exceeds it is wrong even if it "works" on paper.

2. **Idle-RPG, not watchdog.** `require_human_approval=False` by default. The operator may be at dinner; the pipeline must not block them. The PWA is a touch surface for a *curious* operator, not a coercion tool. Gates are an opt-in, not the norm.

3. **Three pillars, each must be real:**
   - **Follow** — see what's happening live (SSE events, phase state, token burn).
   - **Review** — audit what happened (raw-io.jsonl, usage.jsonl, events.jsonl, artifacts).
   - **Intervene** — nudge what's next (operator intents queued → applied/ignored terminal, narrator-refined next-step packet).
   - "Intervene was cosmetic" was true pre-M02B. Keep it real.

4. **Memory palace is a first-class citizen.** Not a footnote. The `wake-up.md`, the `wake_up_hash`, the `memory_refs` — they are the project's long-term brain. Every role, every packet, every usage row should carry provenance from the palace so that "why did this role think X" can always be traced. If a new subsystem cannot answer "what memory informed this decision," it is incomplete. **Status:** aspirational — today usage rows + next-step packets stamp provenance, but `narrative_digest_created` and `discussion_entry_created` events do not. See A-9.

5. **Gamification is the interface, not decoration.** The PWA must feel like an incredible game — tactile, readable at a glance, delightful to poke at. "JRPG aesthetic" is not a skin; it is the whole expressive grammar for surfacing what the SDLC is doing. A screen that looks like a progress bar with color accents has failed the vision. Action ring, object lens, narrator dialogue, timeline ribbon, compose dock, map — these are *the UI*, not ornaments on top of one.

6. **Founding team is prominent and constantly consulted.** When the architect breaks work down, when the micro-task-breaker splits a step, when a gate opens, when a phase fails — the founding team weighs in. They meet. They discuss. Those discussions are documented as `discussion.jsonl` entries + `PERSONA_FORUM.md` updates + narrator digest highlights, and they are reviewable by the operator. If a major decision happens without the founding team being heard from, that is a design bug. The personas are not set-and-forget characters seeded once at project creation — they are recurring voices that shape the run. **Status:** aspirational — today personas are synthesized once at project seed; no subsequent trigger causes a forum consultation. See A-8.

7. **Opencode per role.** Each SDLC role spawns its own opencode subprocess under its own session. No shared pool. Per-project `opencode.json`, routed via `endpoints:` + `routing:` YAML. No residue of the pre-H6 pool/LLMClient machinery.

8. **Deterministic dry-run.** `dry_run=True` auto-wires `*_dryrun` SessionRunners. Not a fake LLM. Real artifact machines that produce coherent output offline. The 500+ tests are valuable because dry-run is real.

9. **Event ledger is the single source of truth.** Every operator-visible action is an event. `EventEngine` on the frontend folds the ledger into game state. Replay = scrub through events. If a feature can't be expressed as ledger events, it doesn't belong.

10. **Narrator is neutral.** JRPG dialogue is UI aesthetic, not narrative identity. The narrator summarizes what happened; it does not roleplay, editorialize, or invent plot. That role belongs to the founding team.

**Roadmap anchor:** `project-management/srs-mini-phases/M01–M12` is the live plan. The `phases/harness-*` set is historical scaffolding (landed). Do not treat harness docs as current.

**What "good" looks like in this codebase:**
- One emitter per contract; three emitters of the same event shape is a smell.
- Declarative tables over stair-step if/elif when the branches are symmetric.
- Engine knows about phases; roles know about themselves; HTTP knows about HTTP. If `control.py` reaches into `PersonaForum._private_method`, that is a layering violation.
- `except Exception: pass` is a bug, not defensiveness.
- Docs are code, not decoration — if `docs/getting-started.md` ImportErrors, the repo is lying.

### 0.2 How this document works

This is a **running checklist**. Every finding has:

- A stable ID (`B-1`, `A-2`, `C-3`, `X-4`) that never renumbers.
- A status box: `[ ]` open · `[~]` in progress · `[x]` done · `[!]` blocked.
- A short description + file:line refs.
- An **Evidence** subsection with explicit slots. When you close an item, you paste the proof into those slots. Commit hash, test output, screenshot, log snippet — whatever the evidence field asks for.

**Ground rules for the agent working this list:**

1. **Update the checklist as you work.** Flip `[ ]` → `[~]` when you pick up an item, `[~]` → `[x]` when you close it. One item in progress at a time per thread.
2. **Never mark an item `[x]` without filling its Evidence block.** An empty Evidence block on a closed item is fraud. If you cannot produce the evidence the slot asks for, the item is not done — keep it `[~]` and say why.
3. **Paste the actual output, not a description.** `"tests pass"` is not evidence. `$ .venv/bin/python -m pytest tests/test_narrator.py -q\n........ 12 passed` is evidence.
4. **If a finding is wrong, say so, and move it to §11 "Rejected findings" with reasoning.** Don't silently delete.
5. **When you discover new issues during a fix, append them** — don't stash them in your head. Use the next free ID in the appropriate section.
6. **Update the headline numbers in §3 whenever you change test counts, line counts, or module shapes.**
7. **Cross-link commits.** When you close a finding, note the commit SHA in the Evidence slot. Onboarders use this to read the diff.
8. **Do not reorganize or renumber.** Other documents (and future agents) reference these IDs.
9. **Full-suite green slot — proportional rigor.** Doc-only fixes, comment edits, and single-line refactors do not need a full `pytest` paste every time; a narrow targeted test is enough, plus one full-suite checkpoint per batch of commits (paste into the batch's final finding). Anything that touches `orchestrator/engine.py`, the event bus, the sidecar, or the PWA fold *does* need a full-suite paste on that item.

**What this document is NOT:**
- Not a plan for new features. Feature plans live in `project-management/srs-mini-phases/`.
- Not a journal. Keep findings factual and file:line-scoped.
- Not a place for abstract discussion. If you want to discuss a design principle, put it in §1 under the relevant constraint, not in a finding.
